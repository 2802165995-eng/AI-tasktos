# TasteOS 分析取消与失败恢复设计

## 背景

当前图片分析流程在开始请求前把 `isAnalyzing` 设置为 `true`，请求结束后再恢复为 `false`。这条路径存在四个问题：

1. `fetch` 没有超时控制，网络连接或上游服务长期无响应时，Promise 会一直处于 pending。
2. `isAnalyzing` 会随整个应用状态写入 `localStorage`，分析过程中刷新页面后会恢复出一个没有对应运行任务的“分析中”状态。
3. 状态清理分散在成功路径末尾，没有由 `finally` 统一保证；未覆盖的异常可能跳过清理。
4. 没有取消控制器和任务身份校验，用户无法主动中止请求，旧请求也可能在新操作之后迟到并覆盖界面状态。

## 目标

- 分析过程中提供明确可用的“取消分析”按钮。
- 取消真实模型请求时真正中止浏览器请求。
- 取消后立即恢复表单，不新增案例，不生成离线回退结果。
- 请求失败或超时后始终退出“分析中”状态。
- 页面刷新后不恢复已经失效的运行中任务。
- 防止已取消或已过期请求的迟到结果修改当前状态。

## 非目标

- 不改动模型提示词、分析 JSON Schema 或供应商配置。
- 不增加队列、并发上传或断点续传。
- 不重构整个 `src/app.js`。
- 不改变普通 API 失败时现有的离线回退策略；只有用户主动取消时不回退。

## 设计

### 请求控制

`analyzeReferenceViaApi` 接受以下可选参数：

- `signal`：由界面层传入的 `AbortSignal`。
- `timeoutMs`：请求最大等待时间，默认 45 秒。

API 客户端创建内部超时控制器，并把外部取消与内部超时合并到发送给 `fetch` 的信号。无论请求成功或失败，都清除超时计时器。

错误语义保持可区分：

- 用户主动取消：抛出可识别的取消错误。
- 达到超时：抛出“模型分析超时，请重试”。
- HTTP、网络或响应格式错误：保留现有错误信息。

### 页面任务状态

模块级维护当前分析任务：

```text
currentAnalysisTask = {
  id,
  controller,
  cancelled
}
```

任务对象不写入 `localStorage`。持久化状态中的 `isAnalyzing` 只用于渲染当前会话；`loadState` 和状态标准化会强制把启动时读取到的遗留 `isAnalyzing` 重置为 `false`。

每次开始分析时创建新任务。异步步骤结束后，只有任务 ID 仍与当前任务一致时才能提交结果或更新提示。这样即使底层请求无法及时停止，迟到结果也不会污染新状态。

### 取消交互

分析期间：

- 原“开始分析”按钮保持禁用并显示“分析中...”。
- 同一区域显示可点击的次要按钮“取消分析”。
- 输入控件继续禁用，避免当前任务的数据源在执行中变化。

点击取消后：

1. 标记任务为已取消。
2. 调用 `AbortController.abort()`。
3. 清除当前任务引用。
4. 设置 `isAnalyzing: false`。
5. 显示“已取消本次分析。”。
6. 不新增 reference，不新增 analysis，不执行离线回退。

### 统一清理

`handleCreateReference` 使用 `try/catch/finally` 管理任务生命周期：

- `try`：读取输入并执行真实或离线分析。
- `catch`：区分用户取消与普通错误；普通 API 错误继续按现有规则回退离线分析。
- `finally`：如果当前任务仍是本任务，清除任务引用并确保 `isAnalyzing` 为 `false`。

本地图片读取发生在任务创建之前。读取失败时直接显示失败提示，不进入分析状态。图片读取若未来被纳入任务期间，也必须走同一清理路径。

## 测试

### API 客户端

- 外部 `AbortSignal` 会传递到 `fetch`。
- 主动取消会产生可识别的取消错误。
- 超时会终止请求并返回明确超时错误。
- 请求完成后会清理计时器。

### UI 与状态

- 分析中渲染“取消分析”按钮及其事件钩子。
- 取消逻辑不会新增案例或分析结果。
- 普通失败仍可回退离线分析，并退出分析状态。
- 从 `localStorage` 读取 `isAnalyzing: true` 时，启动状态被恢复为 `false`。
- 旧任务完成后不能覆盖已取消任务或更新后的状态。

### 手动验证

1. 启动本地服务并选择真实模型分析。
2. 发起分析后点击取消，确认表单立即恢复且没有新增案例。
3. 模拟长时间 pending 的请求，确认 45 秒后显示超时并退出分析状态。
4. 分析过程中刷新页面，确认页面恢复为可操作状态。
5. 正常分析一次，确认真实模型结果仍可保存。

## 验收标准

- 任意失败、超时、取消和刷新路径都不会让界面永久停留在“分析中”。
- 取消按钮在请求期间可见且可操作。
- 用户取消不会生成离线结果。
- 真实模型成功和普通失败回退行为不发生回归。
