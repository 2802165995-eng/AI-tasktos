# TasteOS Design System Skill 设计规范

## 目标

生成可被 Codex 显式调用的 `tasteos-design` Skill，用于把 TasteOS 及同类 AI 视觉创作工具设计为发布级 PC 网站前端。设计语言以 `docs/design/tasteos-workbench-concept.png` 为主要视觉参考，但不逐像素复刻；保留其工作台信息架构，并补齐成熟产品所需的 Token、组件状态、桌面适配、可访问性与极端内容规则。

## 使用边界

- 仅在用户明确要求 “TasteOS style”、“TasteOS design”、`/tasteos-design` 或应用 TasteOS 设计系统时触发。
- 主要平台是 PC 端 Web，不定义移动端 App 形态。
- 设计基准为 1440 × 900，支持 1280–1920px。
- 1200px 以下不承担完整适配；允许提示扩大窗口或进入受控横向滚动。
- 该 Skill 约束视觉与交互表现，不修改产品业务逻辑、数据模型或 API。

## 设计哲学

TasteOS 是视觉研究工作台，不是营销型卡片页。图像承担识别，结构承担效率，颜色只用于状态和动作。界面应在高信息密度下保持可扫描，通过稳定的四区工作台、严格对齐和明确交互状态，呈现成熟大型科技公司的生产级前端质量。

核心张力是“专业工具的密度”与“视觉创作的轻盈感”：结构严谨，但不沉重；有品牌识别，但不炫技。

## PC 工作台架构

页面采用四层横向结构：

1. 顶部全局导航：品牌、核心模块、当前空间、模型状态、用户入口。
2. 左侧功能栏：固定宽度，承载工作流级导航。
3. 案例库面板：固定宽度，包含搜索、筛选和双列缩略图。
4. 中央分析区：弹性主工作区，承载图片、分析结论、提示词与反馈动作。
5. 右侧配置栏：固定宽度，承载上传、模型选择和分析设置。

宽度策略：

- 1440px：完整显示所有区域。
- 1280–1439px：压缩案例库和配置栏，中央区保持最低可读宽度。
- 1200–1279px：默认收起右侧配置栏，通过按钮唤出抽屉。
- 低于 1200px：显示桌面宽度提示，不尝试移动端重排。

## 视觉系统

### 色彩

- 页面背景：冷雾白 `#F5F7FA`
- 主表面：`#FFFFFF`
- 次级表面：`#F8FAFC`
- 深色正文：`#172033`
- 次级正文：`#667085`
- 弱文本：`#98A2B3`
- 细边框：`#E4E8EF`
- 可见边框：`#CBD5E1`
- 工具蓝：`#2F6BFF`
- 蓝色浅底：`#EAF1FF`
- 成功：`#16A36A`
- 警告：`#D97706`
- 错误：`#DC3E42`

仅核心“开始分析”按钮可使用蓝紫红渐变。其他组件不得使用渐变作为常规装饰。

### 字体

- 展示与 UI：Inter；中文回退为 `"PingFang SC", "Microsoft YaHei", sans-serif`
- 参数、提示词片段、时间与模型标识：JetBrains Mono
- 页面标题控制在 24–32px；工具界面不使用营销页式超大标题。

### 几何与层级

- 基础间距使用 8px 网格。
- 小元素圆角 4–6px，按钮与输入框 8px，面板 10–12px，浮层 12px。
- 普通面板以背景和 1px 边框分层，不使用阴影。
- 阴影只用于分析主画布、菜单、弹窗和抽屉。
- 图标采用 Lucide，统一 2px 描边、圆角端点，不混用图标库。

### 动效

- 机械式、精确、无弹跳。
- 微交互 140ms，标准展开 180ms，抽屉与弹窗 200ms。
- 只允许颜色、边框、透明度和 1–2px 位移变化。
- 尊重 `prefers-reduced-motion`。

## 信息层级

分析页面遵循“结论先于解释”：

1. 用途与当前状态
2. 标题与核心风格标签
3. 可复制的反推提示词
4. 提示词基因、色板、负向提示词
5. 详细分析与依据

长文本默认限制可视行数，并提供展开与复制。动作必须紧贴对应结果，避免用户跨页面寻找下一步。

## 组件范围

Skill 至少定义以下组件：

- 顶部导航、左侧功能栏、面板标题栏
- 搜索框、筛选器、分段控制、下拉菜单
- 主按钮、次按钮、幽灵按钮、危险按钮、图标按钮
- 案例缩略卡、分析画布、提示词输出区、配置面板
- 输入框、文本域、上传区、选择框、开关
- 标签、状态徽章、色板、进度条、骨架屏
- 表格、列表行、分页、空状态、错误状态
- Tooltip、Popover、Modal、右侧抽屉、轻量通知

所有交互组件必须覆盖 default、hover、focus-visible、active、disabled；异步组件还需 loading 和 error 状态。

## 核心交互

- 案例卡支持鼠标选择、键盘焦点、快捷删除和明确选中态。
- 上传区支持点击、拖拽和 URL 输入，并在分析前显示预览。
- 右侧配置栏可折叠并记忆用户状态。
- 分析加载只阻塞相关区域，不锁死整页。
- 复制成功优先在按钮附近显示短暂状态，不连续堆叠 Toast。
- 空状态必须给出直接下一步；错误状态必须给出原因和恢复动作。
- 删除等破坏性行为使用二次确认，不把危险动作作为卡片主操作。

## 可访问性与内容韧性

- 正文对比度不低于 WCAG AA。
- 所有操作可由键盘完成，并具有清晰 `focus-visible`。
- 图标按钮必须有可访问名称。
- 不以颜色作为唯一状态信号。
- 测试中文长标题、超长提示词、0 条案例、100 条案例、图片加载失败和 API 失败。
- 图片区域使用稳定尺寸和占位，避免布局跳动。

## 验证标准

- Hue `validate.mjs` 退出码为 0。
- Skill Creator `quick_validate.py` 退出码为 0。
- 四个 HTML 无未替换占位符、孤立 CSS 类或未定义 Token。
- 1440 × 900 和 1280 × 800 浏览器截图中无溢出、遮挡或不可读区域。
- Light/Dark 模式均可渲染；TasteOS 产品本身以 Light 为主模式。
- `app-screen.html` 必须达到真实产品中途使用状态的密度，不得像线框图。

## 非目标

- 不复制其他公司的 Logo、专有图标或受保护资产。
- 不把参考图逐像素复刻。
- 不以玻璃拟态、霓虹光晕、大面积渐变或动画作为视觉卖点。
- 不为移动端重新设计完整信息架构。
