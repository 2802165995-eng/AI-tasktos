# TasteOS 面试追问准备

## 1. 这和直接问 ChatGPT 生成提示词有什么区别？

ChatGPT 更像一次性问答，TasteOS 管的是长期偏好资产。

核心区别：

- TasteOS 有参考图案例库。
- TasteOS 会沉淀个人审美画像。
- TasteOS 按具体场景和模型格式生成提示词。
- TasteOS 有结果反馈，能把失败经验沉淀为后续约束。
- TasteOS 的提示词不是从零生成，而是由用户长期收藏、模型分析和反馈共同驱动。

一句话回答：

> ChatGPT 是一次性生成文本，TasteOS 是把用户长期视觉偏好沉淀成可复用提示词系统。

## 2. 这和 Pinterest / 花瓣有什么区别？

Pinterest 和花瓣解决收藏问题，TasteOS 解决收藏之后如何理解、复用和迭代的问题。

可以这样说：

> Pinterest 停在“我喜欢这些图”，TasteOS 往后走两步：先分析“我为什么喜欢这些图”，再把分析结果变成可复制到生图工具里的提示词资产。

## 3. 为什么不直接做生图工具？

因为第一版要验证的是生图前的关键痛点：用户无法稳定表达审美。

如果一开始接生图能力，项目重点容易变成模型效果和图片质量，而不是产品问题。TasteOS 先解决提示词准备和偏好沉淀，后续接入生图 API 会更自然。

## 4. 现在的视觉分析是真实 AI 吗？

当前支持两种模式：

- 离线演示分析：不需要 API Key，用本地规则保证完整流程可跑。
- 真实模型分析：通过 `/api/analyze-reference` 代理调用 DashScope / OpenAI 视觉模型，返回结构化中文分析。

我会主动说明：离线分析不是模型识图，它是保证演示稳定的可替换分析层；真实模型分析已经接入，但需要本地 `.env.local` 配置 Key。

## 5. 为什么要支持即梦、通义万相、豆包生图？

因为国内用户实际更可能把提示词复制到这些工具里。只输出通用提示词或 Midjourney 格式，会显得离用户真实工作流比较远。

我把提示词生成拆成两层：

- 先生成通用的审美约束和场景化提示词。
- 再通过 `promptAdapters.js` 转成不同模型更容易理解的格式。

这样后续增加新模型格式不需要改主流程。

## 6. Taste Profile 怎么生成？会不会很主观？

设计审美本来有主观性，所以 TasteOS 不做“客观评分”，而做“偏好提取”。

Taste Profile 主要来自：

- 用户导入的参考图。
- 单图视觉分析结果。
- 用户对生成结果的反馈。

它输出的是“用户偏好”，不是“设计好坏判断”。

## 7. 为什么只做海报、社媒封面和作品集封面？

这三类场景有共同结构：

- 都是平面视觉。
- 都有标题、主视觉、信息层级和传播目标。
- 案例容易收集，用户也容易真实使用。

如果第一版扩展到 UI、包装、视频和完整品牌系统，分析维度会变散，MVP 很难验证核心闭环。

## 8. 为什么要做 feedback loop？

提示词不是一次性任务。用户生成图片后，常见问题包括文字不可读、画面太乱、颜色不对、情绪不对。

这些失败经验如果不被记录，下次仍然会重复犯错。TasteOS 把反馈沉淀为负向偏好和提示词约束，让后续生成更稳定。

## 9. 最大不足是什么？

建议主动承认：

- 当前是本地 MVP，没有账号系统和云数据库。
- 真实模型分析依赖用户自己的 API Key。
- 目前不直接生成图片，只输出提示词包。
- 案例数据还可以继续扩充为更真实的设计素材。

补一句：

> 但这些不足不影响第一版验证核心假设：用户是否需要把视觉参考转化为长期可复用的个人提示词系统。

## 10. 如果继续做第二版，你会做什么？

优先级：

1. 图片分析缓存，避免同一张图重复调用模型。
2. 同一提示词在多个生图工具中的结果对比。
3. 浏览器插件，一键保存网页参考图。
4. 云端账号和跨设备同步。
5. 团队版品牌视觉偏好库。
